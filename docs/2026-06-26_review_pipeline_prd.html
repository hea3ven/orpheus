1. Summary

Orpheus should promote local review from an informal operator habit into a first-class, configurable task workflow. After an implementation agent completes a task, the operator should run orpheus task review <task-id>. The selected review pipeline executes read-only review steps, records structured findings, blocks on current-task issues, optionally creates separate follow-up Beads, and publishes/finalizes the task only after the pipeline passes.

The review pipeline itself is the approval mechanism. A default zero-config pipeline contains a manual review step, so Orpheus remains safe by default. Repositories may opt into richer pipelines with deterministic checks and attached AI review agents. If a configured automated-only pipeline passes, automatic PR creation/finalization is intentional.

Target flow

orpheus task run TASK-123
# agent implements the task and runs `orpheus agent done ...`

orpheus task review TASK-123
# review pipeline runs
# blockers stop the pipeline
# passing pipeline invokes the same finalization path as `task done`

# if publication fails after review passed:
orpheus task done TASK-123

2. Problem

Current Orpheus workflows distinguish agent completion from PR/merge state, but local review is still too implicit. Operators must remember how to inspect the right worktree, run any checks, decide whether the agent output is acceptable, and then run task done. This leaves several gaps:

  • No explicit approval contract: task done can publish completed agent work without a recorded local review result.
  • No structured review findings: AI, check, and manual review feedback are not captured in Orpheus state.
  • No safe follow-up loop: Re-running an agent after review should address recorded blockers, not reimplement the original task.
  • No configurable review policy: different repos need different checks, review agents, and manual review affordances.
  • No read-only enforcement: review commands and review agents should not mutate candidate changes.
  • No path for review-generated follow-up tasks: nonblocking findings that should become separate work need human-approved Beads creation.

3. Goals and non-goals

Goals

  • Make task review the required local approval gate before task done can publish/finalize.
  • Support typed review pipelines with check, agent_review, and manual steps.
  • Record review attempts, executed steps, findings, waivers, and separate-task creation results in Orpheus task state.
  • Keep review steps read-only through per-step snapshot/restore enforcement.
  • Let review agents record findings through an Orpheus CLI contract instead of provider-specific JSON schemas.
  • Let task run infer follow-up mode from blocked review state, with no --follow-up flag.
  • Support repo-specific review pipeline selection through the Orpheus registry.
  • Keep the initial manual review UX simple: terminal prompts, no TUI investment.

Non-goals for v1

  • No PR-comment response loop after a PR already exists.
  • No background/detached review agent supervision.
  • No provider-specific structured output schema or codex exec --output-schema integration.
  • No capture of full check or agent logs.
  • No exact final snapshot enforcement at task done; the user remains responsible for edits after review passes.
  • No task done --skip-review or force bypass.
  • No check waivers in v1.
  • No inline repo-local pipeline definitions.
  • No empty review pipelines.
  • No no-change finalization path.

4. Core concepts

Candidate changes

The candidate is the current uncommitted Git-visible state that Orpheus will eventually commit through task done. It includes tracked modifications, tracked deletions, file mode changes, and untracked non-ignored files. Ignored files are out of scope.

Review pipeline

A named ordered list of review steps. The pipeline is the approval policy: if the pipeline passes, Orpheus may publish/finalize automatically.

Review attempt

One execution of task review for a task. Every invocation starts a fresh attempt. Older attempts are audit history only; the latest attempt is authoritative for status and follow-up behavior.

Review step

One executed step in a review attempt. Step kinds are check, agent_review, and manual.

Finding types

Type Meaning Blocks?
blocking Must be fixed in the current task before publication, unless explicitly waived or converted. Yes
advisory Useful note that does not block the current task. No
separate-task Nonblocking candidate for a new standalone Bead. No

Operational failure vs code blocker

A code blocker is normal review feedback: failing tests, AI review blockers, or manual blockers. An operational failure means the review process itself failed: invalid config, review agent non-zero exit, process start failure, mutation by a read-only step, or restore failure. Code blockers lead to follow-up implementation. Operational failures require operator attention.

5. Configuration

Global config shape

This feature may introduce a breaking config schema. The project is early enough that avoiding legacy schema debt is preferred. 

agents:
  defaults:
    implementer: gpt-medium
    reviewer: gpt-smart

  profiles:
    gpt-medium:
      harness: codex
      model: gpt-5.5
      effort: medium

    gpt-smart:
      harness: codex
      model: gpt-5.5
      effort: high

reviews:
  default_pipeline: standard
  pipelines:
    standard:
      steps:
        - kind: check
          name: test
          command: make
          args: ["test"]

        - kind: agent_review
          name: ai-review

        - kind: manual
          name: local-review

Agent profile decisions

  • agents.defaults.implementer is used for implementation and review follow-up runs.
  • agents.defaults.reviewer is used for agent_review steps unless a step overrides the agent.
  • harness and model are generic profile fields.
  • effort is harness-specific but kept at the profile top level for readability.
  • No extra_args in v1.
  • harness: codex assumes the codex executable; executable-path configuration is deferred.
  • Implementation and follow-up runs remain attached/interactive for v1.
  • Review agents also use the attached launcher/profile mechanism for v1.

Review pipeline resolution

Pipeline selection precedence:

  1. orpheus task review --pipeline <name>
  2. Repo registry field review_pipeline
  3. Global reviews.default_pipeline
  4. Built-in default manual pipeline

Repo registry example:

repos:
  - id: orpheus
    path: /home/mati/Documents/prog/orpheus
    review_pipeline: go-standard

Built-in default pipeline

If no configured pipeline is available, Orpheus uses a single manual step with no command:

steps:
  - kind: manual
    name: local-review

Pipeline validation

  • Pipelines must contain at least one step.
  • Step names must be non-empty for readable output.
  • Step names do not need to be unique in v1 because they are not semantic IDs.
  • Commands run directly without shell parsing. Users can explicitly configure bash -lc if they need shell behavior.
  • No interpolation tokens in check/manual commands for v1; review environment variables are passed instead.
  • Repos may select named global pipelines only; no inline per-repo pipelines in v1.

6. Workflow

Normal implementation and local review

orpheus task run TASK-123
agent runs `orpheus agent context`
agent edits files
agent runs `orpheus agent done --summary ... --description ... --detailed-description ...`

orpheus task review TASK-123
  preflight finalization readiness
  require candidate changes
  require clean real Git index
  run review pipeline
  offer creation of separate-task candidates
  mark review passed
  call internal task done finalization path

Blocked review follow-up

orpheus task review TASK-123
  step reports blocking findings
  pipeline stops with review status blocked

orpheus task run TASK-123
  Orpheus infers follow-up mode from latest blocked review
  agent context says: address only these findings
  targeted findings are marked as targeted by this run

orpheus task review TASK-123
  starts a fresh authoritative review attempt

Publication failure after review passes

orpheus task review TASK-123
  review passes
  task done finalization fails, e.g. gh auth failure

# Review remains passed. User fixes publication issue, then retries:
orpheus task done TASK-123
Decision: review is independent of finalization Review attempts record review outcomes only. Publication/finalization failures belong to the finalization path and related task-state events, not to review attempt fields. A passed review remains passed after publication failure so the operator can retry task done without rerunning review.

7. Review step behavior

Common step execution rules

  • Before each step, Orpheus snapshots the Git-visible candidate state.
  • After each step, Orpheus compares the candidate state.
  • If the step mutated files, Orpheus restores the snapshot automatically.
  • A mutation by any review step is an operational failure, even if restore succeeds.
  • If restore fails, the task projects to Needs attention and the user must inspect the repo manually.
  • Snapshot/restore covers tracked changes, tracked deletions, mode changes, and untracked non-ignored files.
  • Ignored files and external side effects are not guaranteed to be restored.
  • Stdout/stderr stream live to the terminal. No logs are captured in v1.
  • Review env is passed to all steps.

Review environment

ORPHEUS_REPO_ID=...
ORPHEUS_TASK_ID=...
ORPHEUS_AGENT_PURPOSE=review
ORPHEUS_REVIEW_ATTEMPT=1
ORPHEUS_REVIEW_STEP=ai-review

ORPHEUS_REVIEW_STEP is informational. agent review add validates the attempt and active current step kind, not the step name.

Check steps

- kind: check
  name: test
  command: make
  args: ["test"]
  • Process start failure, such as executable not found, is an operational failure.
  • Exit code 0 passes.
  • Non-zero exit creates a blocking finding.
  • Check blockers are not waivable in v1.
  • No stdout/stderr log capture in v1; the fixing agent is expected to rerun the command and inspect output.

Agent review steps

- kind: agent_review
  name: ai-review
  agent: gpt-smart # optional; defaults to agents.defaults.reviewer
  • Uses the same attached launcher/profile mechanism as implementation agents.
  • Uses the bootstrap prompt pattern: the agent is instructed to run orpheus agent context.
  • orpheus agent context renders review-specific context when ORPHEUS_AGENT_PURPOSE=review.
  • The context includes task details, completion summary, review instructions, the read-only contract, and exact orpheus agent review add usage.
  • The review agent is told to review current working-tree changes and may use Git commands as needed.
  • The agent records findings incrementally through orpheus agent review add.
  • No agent review done command exists in v1. Agent process exit is enough.
  • Exit 0 with no unwaived blockers passes.
  • Non-zero exit is an operational review failure, even if partial findings were recorded.

Manual steps

- kind: manual
  name: local-review
  command: git      # optional
  args: ["diff"]   # optional

At the start of every manual step, Orpheus prints context:

  • task ID and title;
  • agent completion summary;
  • git status --short;
  • git diff --stat.

If a manual command is configured, Orpheus runs it after printing context. Then it prompts:

Manual review decision for TASK-123:
  1. Approve and continue pipeline
  2. Record blocking finding
  3. Record advisory finding
  4. Record separate-task candidate
  5. Abort review

Manual findings are collected by Orpheus prompts, not by asking the user to call orpheus agent review add. Manual commands still receive review env for custom scripts.

8. Findings, blockers, waivers, and follow-up tasks

Finding schema

type: blocking | advisory | separate-task
title: string
description: string
suggested_action: string # required for blocking, optional otherwise
waiver:
  reason: string
  at: timestamp
task_proposal:
  title: string
  description: string
  acceptance_criteria: string
created_task_id: string

Blocking findings

  • Blocking findings stop the pipeline unless waived or converted.
  • Agent-review blockers can be waived by the human with a reason.
  • Manual blockers are controlled by what the user records.
  • Check blockers cannot be waived in v1.
  • Waivers are scoped to the review attempt and do not carry to future attempts.

Converting blockers to separate-task candidates

The user may convert an agent-review blocker into a separate-task candidate when it is not required by the current task acceptance criteria. Conversion requires a task proposal or user-provided proposal fields and a reason. Once converted, it is nonblocking for the current attempt.

Separate-task candidates

  • separate-task findings are nonblocking by definition.
  • After all blockers are cleared, Orpheus offers to create Beads for selected candidates before publication.
  • Created Beads are standalone tasks with provenance in their description, not dependencies or children by default.
  • Created task IDs are recorded back on the finding.
  • If creation fails, Orpheus stops by default and asks whether to continue publication anyway.
  • If the user declines to create candidates, review can still pass and publish.
  • No cross-attempt duplicate detection in v1. Within one attempt, already-created candidates are not offered again.

9. Task-state model

Review attempts

review_attempts:
  - attempt: 1
    status: blocked
    pipeline: go-standard
    started_at: 2026-06-26T10:00:00Z
    finished_at: 2026-06-26T10:12:00Z
    steps:
      - kind: check
        name: test
        command: make
        args: ["test"]
        exit_code: 2
        failed: false
        findings:
          - type: blocking
            title: "Check failed: test"
            description: "Command `make test` exited with status 2."
            suggested_action: "Run the check, inspect the output, fix it, then rerun task review."

  - attempt: 2
    status: passed
    pipeline: go-standard
    steps:
      - kind: agent_review
        name: ai-review
        agent: gpt-smart
        failed: false
        findings: []
      - kind: manual
        name: local-review
        failed: false
        findings: []

Attempt statuses

Status Meaning
running A review pipeline is currently executing.
blocked The pipeline stopped on valid code/product blockers.
failed The review process failed operationally.
passed The review process completed successfully and publication is allowed.
aborted The user aborted manual review, or the attempt was superseded by a new review run.

Current step

While a review attempt is running, task state persists a current_step so orpheus agent review add can append findings from a separate process. 

review_attempts:
  - attempt: 3
    status: running
    pipeline: go-standard
    current_step:
      kind: agent_review
      name: ai-review
      agent: gpt-smart
      failed: false
      findings: []
    steps:
      - kind: check
        name: test
        command: make
        args: ["test"]
        exit_code: 0
        failed: false
        findings: []

When the step completes, Orpheus moves current_step into steps and clears it. Historical attempts should not retain an active current_step.

State management decisions

  • Review attempt counters are separate from agent run attempt counters.
  • Attempt numbers are enough for v1; no opaque IDs.
  • Every task review invocation creates a new attempt.
  • Any previous running attempt is marked aborted and its current step is finalized into steps.
  • The latest review attempt is authoritative. Older attempts are audit history only.
  • No cross-attempt matching or resolution of findings.

10. Command behavior

orpheus task review <task-id>

  • Resolves the task and selected review pipeline.
  • Runs a finalization preflight using the same readiness checks as task done.
  • Refuses known unsupported finalization states before running the pipeline, such as an existing PR URL.
  • Requires candidate changes if there is no recorded finalization commit.
  • Requires a clean real Git index before review starts.
  • Starts a new review attempt and aborts any previous running attempt.
  • Runs steps in configured order.
  • Handles blockers, waivers, conversions, and separate-task candidates.
  • Marks the attempt passed only after nonblocking follow-up handling is complete.
  • Calls the same internal finalization path as task done without shelling out to the CLI.

orpheus task done <task-id>

  • Requires the latest review attempt to have status passed.
  • Refuses direct publication after agent done if review has not passed.
  • Refuses if the latest review has open blockers.
  • Does not support --skip-review in v1.
  • Does not compare a final candidate snapshot in v1; edits after review are the user's responsibility.
  • Records finalization/publication failures through the finalization path or task-state events, not on review attempts.

orpheus task run <task-id>

task run infers implementation vs follow-up from task state.

State task run behavior
No prior completion / normal runnable task Implementation run using agents.defaults.implementer unless overridden.
Latest review is blocked with untargeted blockers Follow-up run. Context includes blockers and instructs the agent to address only those findings.
Blockers were already targeted by a follow-up run and no new review happened Refuse; require task review again.
Latest review is aborted, failed, or passed Refuse; next action is review retry, operational fix, or publication.

orpheus agent context

  • Becomes purpose-aware through ORPHEUS_AGENT_PURPOSE.
  • Missing purpose defaults to implementation for compatibility inside normal task run.
  • Review context requires review env and renders review-specific instructions.
  • Review context includes task context and completion summary.
  • Review context instructs the agent to inspect current working-tree changes using Git commands as needed.
  • Review context documents the exact agent review add contract.

orpheus agent review add

orpheus agent review add \
  --type blocking \
  --title "Missing validation for empty ID" \
  --description-file finding.md \
  --suggested-action "Add validation and tests"
orpheus agent review add \
  --type separate-task \
  --title "Duplicate validation helper" \
  --description-file finding.md \
  --task-title "Extract shared validation helper" \
  --task-description-file task.md \
  --task-acceptance-criteria-file acceptance.md

Validation

  • Requires active review env.
  • Requires ORPHEUS_REVIEW_ATTEMPT to match the latest running attempt.
  • Requires current_step.kind == agent_review.
  • Appends findings directly to current_step.findings.
  • No edit/remove support in v1.
  • Exactly one --type: blocking, advisory, or separate-task.
  • --description and --description-file are mutually exclusive; exactly one is required.
  • blocking requires --suggested-action.
  • separate-task requires task proposal fields.

11. Status projection

Status groups should distinguish normal workflow state from operational problems. Needs attention is reserved for errors, inconsistent state, or operational failures. Normal review blockers are not errors.

State Group Detail
Agent completed, no review yet Reviewing local review; run task review
Latest review blocked by code/product findings Idle review blocked by N finding(s); run task run
Follow-up run completed after targeting blockers Reviewing follow-up complete; run task review
Latest review aborted Reviewing review aborted; run task review
Latest review failed operationally Needs attention review failed: <reason>
Review passed, publication failed Needs attention review passed; publication failed; run task done
PR exists Reviewing PR URL

12. Requirements

R1. Review pipeline config

Orpheus must load and validate named review pipelines from global config and resolve repo-specific pipeline selection from the registry.

R2. Built-in safe default

If no review pipeline is configured, task review must use a built-in single manual step with no command.

R3. Review attempt persistence

Task state must persist review attempts, attempt status, pipeline name, current step while active, executed steps, findings, waivers, and created follow-up task IDs.

R4. Review gate for task done

task done must refuse to publish/finalize unless the latest review attempt passed.

R5. Preflight before review

task review must run the same finalization readiness checks as task done before executing the review pipeline, and again before final publication.

R6. Candidate changes and index policy

task review must require candidate changes and a clean real Git index before starting.

R7. Read-only step enforcement

Every review step must be wrapped in snapshot/restore checks. Any mutation must be restored automatically and treated as operational failure.

R8. Check steps

Check steps must run direct commands, stream output live, create blocking findings on non-zero exit, and treat process start failures as operational failures.

R9. Agent review steps

Agent review steps must run attached reviewer agents, render review-specific context, and record findings through agent review add.

R10. Manual steps

Manual steps must print review context, optionally run a configured command, and collect approve/block/advisory/separate-task/abort decisions through simple prompts.

R11. Follow-up run inference

task run must infer follow-up behavior from latest blocked review state, without requiring a --follow-up flag.

R12. Separate-task candidate creation

After a pipeline has no blockers, Orpheus must offer to create selected separate-task candidates as standalone Beads and record created task IDs.

R13. Status projection

Status must project normal review blockers to Idle, review boundaries to Reviewing, and operational review/finalization failures to Needs attention.

13. Deferred future ideas

Provider-specific structured review output

Codex and other agents may support schema-constrained output, such as codex exec --output-schema. This is useful later, but v1 should use the Orpheus CLI contract so review findings work with the existing raw command/args profile model and attached agents.

Background review agents

Running review agents non-interactively in the background remains valuable, but it requires process supervision, logs, stale process recovery, and possibly provider-specific harnesses. V1 keeps review attached and visible.

PR comment response loop

Future workflow: PR comments detected, task run addresses comments, task review reviews the update, and Orpheus pushes to the existing PR branch. V1 refuses local review once orpheus.pr_url is set through existing finalization readiness checks.

Exact review-to-publication snapshot enforcement

A future version may store a candidate tree hash at review pass and require task done to publish exactly that tree. V1 enforces read-only review steps but trusts the user after review passes.

Commit immediately after agent completion

Durable local commits after agent done, possibly squashed before PR publication, were considered. This improves recovery but adds branch/history complexity. V1 reviews uncommitted candidate changes and commits only after review passes.

Richer manual review UX

A TUI or editor-based structured findings form could improve ergonomics. V1 uses simple terminal prompts to keep the feature focused.

Check logs

Capturing full stdout/stderr logs into task-state data files is useful later. V1 streams output live and records only command/exit metadata in task state.

Repo-local and inline pipelines

Team-shared repo-local Orpheus config or inline per-repo pipeline definitions may be useful later. V1 keeps reusable global pipeline definitions and repo registry selection.

Sandboxed review execution

Disposable copied worktrees or stronger sandboxes could isolate review steps more completely. V1 runs in-place with snapshot/restore for simplicity and path/tooling fidelity.

Force publish / skip review

A task done --force or --skip-review escape hatch is intentionally deferred. Users can fall back to lower-level Git/Beads operations for exceptional cases.

No-change task finalization

Agents may report completion with no candidate changes. V1 allows agent done to record that claim, but task review refuses because there is no reviewable artifact. An explicit no-change close workflow is deferred.

14. Suggested implementation slices

These are product slices, not approved Beads. They are included to make the PRD easier to turn into implementation work later.

  1. Manual review gate MVP: add review attempts, built-in manual pipeline, task review, task done review-gate enforcement, and status projection for passed/aborted/blocked states.
  2. Configurable pipelines and repo selection: add global pipeline config, registry review_pipeline, CLI --pipeline, and validation.
  3. Check steps: add direct command execution, live output, non-zero blocking findings, and operational start-failure handling.
  4. Read-only enforcement: add per-step snapshot/restore for tracked and untracked non-ignored candidate changes, with mutation failure status.
  5. Agent review CLI contract: add purpose-aware agent context, attached review agent step execution, and agent review add.
  6. Review follow-up runs: make task run infer follow-up mode from blocked review attempts and target findings by run.
  7. Separate-task candidates: add task proposal fields, human-selected Beads creation, provenance text, failure handling, and created-task ID persistence.
  8. Hardening and docs: cover stale running attempt invalidation, finalization-failure retry, examples, and end-to-end tests.
