Cost investigation / Codex harness → Pi harness

The cost rose in review, not in implementation.

A three-day before/after analysis of Orpheus tasks and 290 execution records finds a 50.5% increase in estimated API-equivalent cost after the July 16 harness switch. The dominant mechanism was extra reviewer model turns replaying cached context.

Equal windows Pre: 13–15 July
Post: 16–18 July
Cutover: 16 Jul, 08:47 CEST
Model: GPT-5.5 / high thinking
Executive verdict

The switch is associated with a real cost increase, but “Pi makes every run more expensive” is false. Implementation cost per execution fell 7%. Review cost per execution rose 49%, review executions rose 30%, and 73% of the total dollar increase came from additional cached-input tokens.

06 / Conclusions

What the evidence supports.

Confidence reflects the strength of the local evidence, not statistical certainty from a randomized experiment.

  1. The increase is real in Orpheus's estimate High confidence

    Equal windows show +50.5% cost and +67.1% tokens. All 157 priced Pi executions reproduce exactly under the same GPT-5.5 formula used for Codex; the harness did not silently select a more expensive pricing row.

  2. Reviewer interaction is the primary mechanism High confidence

    Review accounts for 75.6% of the net increase. Pi review sessions made 85.6% more model turns but 9.6% fewer tool calls, halving tool calls per turn. This repeatedly replayed cached context.

  3. Task mix and repair intensity amplified the change High confidence

    The post window had more review executions, a lower first-pass approval rate, more repairs per reviewed cohort, and fewer resolutions. Several broad workflow/diagnostic tasks required 3–7 repair runs each.

  4. Pi is not uniformly more expensive High confidence

    Implementation token volume per session was flat and implementation cost per execution fell 7%. The cross-boundary server task also had materially cheaper Pi implementation runs.

  5. The exact causal share needs a matched A/B test Medium confidence

    The windows contain different tasks and evolving review code. The session mechanics strongly implicate the harness, but observational data cannot cleanly separate Pi behavior from harder diffs and stricter review outcomes.

07 / Actions

Measure round trips, then reduce them.

The best next steps preserve review quality while targeting the specific source of cost: model round trips over repeated context.

01

Add turn and tool-call telemetry

Persist model-turn count, tool-call count, and cached-token share per execution, grouped by purpose and profile. Cost totals alone hide the batching failure.

02

Prompt Pi reviewers to batch independent reads

Explicitly ask both reviewer profiles to issue independent inspection/search calls together. Pi supports multiple tool calls in one response; the logs show it often does not exploit that capacity.

03

Run a matched review A/B

Give Codex and Pi the same frozen diff and review prompt, 5–10 repetitions each. Compare findings, turns, cached tokens, cost, and false-positive rate separately for general and architecture review.

04

Track quality-adjusted cost

Do not optimize solely for dollars. Report cost per passed review, blocking findings confirmed by fix, first-pass approval, repair cycles, and resolution so cheaper but weaker review is visible.

05

Set a reviewer round-trip budget

Warn when a short review exceeds a turn threshold without a proportional increase in tool calls or findings. This is safer than lowering the existing repair-loop cap globally.

06

Keep telemetry caveats visible

Separate harness-reported estimates, Orpheus-calculated estimates, and invoice/subscription cost. Keep unknown execution coverage beside every cost view.

08 / Method

Sources, normalization, and limits.

This is a reconstruction from current local state and raw session logs, not a vendor invoice reconciliation.

Telemetry discontinuity

Pi usage initially failed to parse because usage lives under message.usage. The July 18 fix and Doctor recovery backfilled 157 Pi executions. All recovered records have capture timestamps on or after July 18.

Coverage

Pre: 125/128 executions have usage and cost. Post: 158/162. Unknown failed/unmatched executions remain in execution counts but contribute neither tokens nor cost, slightly understating both windows.

Token semantics

Codex records cached input as a subset of input; Pi records uncached input and cached input separately. The analysis normalizes these before decomposition. Total-token values are comparable after normalization.

Local evidence used

  • orpheus task stats --group daily, plus consumption, throughput, implementation, and review views across all registered repositories.
  • Task execution state under ~/.local/share/orpheus/repos/*/tasks/*.yaml, including runs, review steps, usage, cost, findings, repair links, and resolution events.
  • Raw Codex sessions under ~/.codex/sessions and Pi sessions under ~/.pi/agent/sessions.
  • Beads records for exact task titles and resolution dates.
  • Current config and timestamp at ~/.config/orpheus/config.yaml.
  • Orpheus source and Git history, especially Pi capture commit aa1a0d4 (15 Jul) and nested-usage fix b76528d (18 Jul).

Important limitations

  • Costs are estimated API-equivalent costs. They may not equal Codex subscription usage, credits, or a vendor invoice.
  • The comparison is observational. Task complexity, review strictness, and Orpheus code changed at the same time as the harness.
  • Execution-day cost and resolution-day throughput use different anchors; window efficiency ratios should not be read as exact per-task attribution.
  • Task state reflects Doctor recovery performed after the original runs. That improves completeness but means the report is a reconstructed view.
